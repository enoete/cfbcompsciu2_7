üí° What is an Operating System?

Definition: An Operating System (OS) is a collection of system software that manages computer hardware and software resources and provides common services for computer programs. It acts as an intermediary between users and the computer hardware.

üé≠ The Restaurant Analogy

Think of an operating system like the manager of a busy restaurant:

  • The kitchen is like the CPU - where all the actual work (cooking) happens
  • The tables are like memory - limited space that needs to be managed
  • The waiters are like I/O devices - taking orders and delivering food
  • The manager (OS) coordinates everything - deciding which orders to cook first, which tables to seat customers at, and ensuring everyone gets served efficiently

üìú Evolution of Operating Systems

Operating systems have evolved dramatically from simple batch processing systems to sophisticated multi-user environments. Understanding this evolution helps us appreciate the complexity and capabilities of modern operating systems.

1950s - No Operating System

Early computers had no OS. Programmers interacted directly with hardware using machine language. Each program had complete control of the machine.

1950s-1960s - Batch Processing Systems

Jobs were collected in batches and processed sequentially. Operators would load programs from punch cards, and the computer would execute them one after another without user interaction.

1960s - Multiprogramming Systems

Multiple programs loaded in memory simultaneously. When one program waited for I/O, the CPU would switch to another program, improving efficiency.

1970s - Time-Sharing Systems

Multiple users could interact with the computer simultaneously. Each user got a small time slice of CPU time, creating the illusion of dedicated access.

1980s-Present - Personal & Networked Systems

GUIs, networking, multi-processing, and distributed systems. Modern OS support multiple processors, virtual machines, and cloud computing.

Types of Operating Systems

üì¶ Batch Processing Systems ‚ñº

‚è±Ô∏è Real-Time Systems ‚ñº

Definition: Real-time operating systems (RTOS) guarantee response within strict time constraints. They are designed for applications where timing is critical.

üöë The Emergency Room Analogy

Two Types:

  • Hard Real-Time: Missing a deadline is catastrophic (aircraft control, medical devices, nuclear reactors)
  • Soft Real-Time: Missing a deadline degrades performance but isn't catastrophic (video streaming, gaming)

Examples: Aircraft autopilot, anti-lock braking systems (ABS), pacemakers, industrial robots

  • Reduced waiting time

üîÄ Multiprocessing Systems ‚ñº

üè≠ The Factory Assembly Line Analogy

Types:

  • Symmetric Multiprocessing (SMP): All processors are equal and share memory
  • Asymmetric Multiprocessing: One master processor controls others

Benefits: Increased throughput, reliability (if one CPU fails, others continue), better performance for parallel tasks

Multitasking

Definition: Multitasking is a feature of most modern operating systems that permits a single CPU to execute what appears to be more than one program at a time. There are different ways to accomplish multitasking.

üé≠ The Theater Director Analogy

Think of multitasking like a theater director managing multiple actors on stage. Although the director can only focus on one actor at a time, they switch attention so quickly between actors that the entire play flows smoothly. Context switching allows several programs to reside in memory but only one to be active at a time. The active program is said to be in the foreground. The other programs in memory are not active and are said to be in the background. Instead of quitting a program, simply switch the programs between background and foreground.

‚öôÔ∏è What Happens During a Context Switch:

  1. Save State: CPU saves the current process's state (registers, program counter, memory pointers) to memory
  2. Update Status: Mark current process as "Ready" or "Waiting"
  3. Select Next: Scheduler picks the next process to run
  4. Restore State: CPU loads the saved state of the selected process
  5. Resume Execution: Continue from where the process left off

‚è±Ô∏è Cost: Context switching takes time (typically microseconds), which is called "overhead". Too many context switches can slow down the system!

Cooperative multitasking is like a polite conversation where each person voluntarily passes the microphone to others when they're done speaking. It works great when everyone is courteous, but if one person refuses to give up the mic, everyone else has to wait!

‚ö†Ô∏è The Problem with Cooperative Multitasking:

If a process enters an infinite loop or crashes, it never yields control. The entire system can freeze because other processes can't get CPU time!

  • Poor responsiveness if processes don't yield often enough

Interrupts

Definition: An interrupt is an electronic signal to the processor for immediate attention. A processor receiving an interrupt will stop what it is doing and service the interrupt before returning to what it was doing before.

üìû The Phone Call Analogy

Imagine you're writing an essay (running process). Your phone rings (interrupt). You:

  1. Stop writing and note where you were (save state)
  2. Answer the phone (handle interrupt)
  3. Finish the call (complete interrupt handler)
  4. Return to your essay exactly where you left off (restore state)

How Interrupts Work:

  1. Interrupt signal is received
  2. CPU saves current process state to PCB
  3. CPU jumps to Interrupt Service Routine (ISR)
  4. ISR handles the interrupt
  5. CPU restores saved state and resumes process

Four Basic Types of Interrupts

üî• Hardware Interrupts ‚ñº

Definition: Hardware interrupts are immediate priority calls for action such as the activation of a reset button or a serious hardware malfunction, such as a power supply failure.

Examples:

  • Reset button pressed
  • Power supply failure
  • Hardware component malfunction
  • Critical system errors

üö® Highest Priority: Hardware interrupts have the highest priority because they often indicate critical system failures that require immediate attention!

‚å®Ô∏è Input/Output Interrupts ‚ñº

Definition: Input/output interrupts come from peripherals such as a keyboard when it sends a signal to the processor that it requires immediate attention, for example a key stroke has been detected.

Examples:

  • Key pressed on keyboard
  • Mouse movement or click
  • Data arrived on network card
  • Disk read/write completed
  • Printer finished printing

Why It Matters: I/O interrupts are asynchronous - they can happen at any time, so the processor must be ready to handle them quickly to maintain system responsiveness.

üéπ The Piano Teacher Analogy

Imagine a piano teacher (CPU) working on a lesson plan (current process). When a student (I/O device) plays a note (interrupt), the teacher must immediately listen (handle interrupt) to provide feedback, then return to the lesson plan.

üíª Software Interrupts ‚ñº

Definition: Software interrupts are usually traps or exceptions that flag errors in the program execution.

Examples:

  • Division by zero error
  • Accessing invalid memory address (segmentation fault)
  • Stack overflow
  • Invalid instruction
  • System calls (programs requesting OS services)

Types of Software Interrupts:

  • Traps: Intentional interrupts for system calls (like print(), open file)
  • Exceptions: Unexpected errors (like divide by zero, null pointer)

üêõ Debugging Connection: When your program crashes with an error message, that's a software interrupt! The CPU detected an illegal operation and interrupted execution to report it.

‚è∞ Timer Interrupts ‚ñº

Definition: Timer interrupts are scheduled events that the timers trigger, such as an application that needs data read from a sensor every 5 seconds.

Examples:

  • Time-slice expiration (for preemptive multitasking)
  • Scheduled tasks (read sensor every 5 seconds)
  • System clock updates
  • Timeout detection
  • Watchdog timers

Critical Role: Timer interrupts are essential for preemptive multitasking! The OS sets a timer, and when it expires, an interrupt forces a context switch - this is how the OS prevents any single process from monopolizing the CPU.

‚úì The Heartbeat of the OS: Timer interrupts are like the heartbeat of the operating system, ensuring fair CPU distribution and keeping the system responsive!

Interrupt Handlers: FLIH and SLIH

Operating Systems manage interrupts using interrupt handlers that are divided into two parts:

‚ö° First-Level Interrupt Handler (FLIH) - Fast Interrupts ‚ñº

Purpose: A FLIH introduces a jitter to the process execution and mask interrupts. This is undesirable so a programmer will attempt to minimize the execution time of an FLIH, and pass control to the SLIH as soon as possible.

Characteristics:

  • Executes IMMEDIATELY when interrupt occurs
  • Very short execution time (microseconds)
  • Blocks other interrupts while running
  • Does minimal processing - just enough to handle the urgent part
  • Schedules SLIH for detailed processing

What FLIH Does:

  • Acknowledge the interrupt
  • Save critical state
  • Determine interrupt source
  • Queue work for SLIH
  • Re-enable interrupts

üö® The Fire Alarm Analogy

When a fire alarm goes off (interrupt), the FLIH is like immediately:

  • Stopping what you're doing
  • Noting the alarm location
  • Calling the fire department

This takes just seconds. The actual fire response (SLIH) happens later with more resources.

üîÑ Second-Level Interrupt Handlers (SLIH) - Slow Interrupts ‚ñº

Purpose: SLIHs may have a long-lived execution time therefore, SLIHs typically are scheduled similarly to threads and processes.

Characteristics:

  • Executes AFTER FLIH completes
  • Can have longer execution time
  • Scheduled like normal threads/processes
  • Does the detailed, time-consuming work
  • Can be interrupted by FLIHs

What SLIH Does:

  • Process received data packets
  • Update data structures
  • Perform complex calculations
  • Communicate with other processes
  • Complete the interrupt handling

‚öñÔ∏è The Trade-off: Interrupts require the processor to stop and start processes which may also introduce a jitter. However, there are also problems such as latency, the time the system responds to an interrupt, and additionally deadlocking which is when an interrupt holds a resource, preventing use of the resource by another process, whilst anticipating another resource becoming available.

A deadlock occurs when two or more processes are waiting indefinitely for resources held by each other.

Four Conditions for Deadlock (All must be present):

  1. Mutual Exclusion: Resources cannot be shared
  2. Hold and Wait: Process holds resources while waiting for others
  3. No Preemption: Resources cannot be forcibly taken away
  4. Circular Wait: Circular chain of processes waiting for resources

Deadlock Resolution Strategies:

Prevention

Ensure at least one of the four conditions cannot occur

Avoidance

Use algorithms (like Banker's Algorithm) to avoid unsafe states

Detection

Allow deadlocks but detect and recover from them

Recovery

Terminate processes or preempt resources to break deadlock

Definition: Context switching is the mechanism that allows several programs to reside in memory but only one to be active at a time. When switching from one process to another, the OS saves the current process state and loads the saved state of the next process.

‚ö° What Happens During Context Switching:

  1. Save current process state to its PCB (Program Counter, registers, memory maps)
  2. Select next process to run (using scheduling algorithm)
  3. Load saved state of the next process from its PCB
  4. Resume execution of the new process

Cost: Context switching isn't free! It takes time (typically microseconds) and uses CPU cycles. Too much switching = overhead!

Types of Multitasking

ü§ù Cooperative Multitasking ‚ñº

Definition: Cooperative multitasking allows for the sharing of the CPU in that, whenever the CPU becomes 'idle' (when a foreground program is no longer using the CPU), a background program uses the CPU during this idle time.

üé§ The Polite Meeting Analogy

In a cooperative meeting, each speaker talks as long as they want, then voluntarily passes the microphone to the next person. The speaker decides when they're done - no one interrupts them.

Problem: What if someone never gives up the microphone? Everyone else waits indefinitely!

How it works:

  • A process runs until it voluntarily yields control
  • Processes must be "well-behaved" and regularly give up CPU
  • No preemption - OS cannot force a switch

Advantages: Simpler to implement, less context switching overhead, processes can finish critical sections

Disadvantages: One misbehaving program can freeze the entire system, poor response time

üíÄ Historical Note: Early versions of Windows (3.x, 95) and Mac OS (before OS X) used cooperative multitasking. This is why one frozen program could crash the whole system!

‚è∞ Time-Slice Multitasking (Preemptive) ‚ñº

Definition: Time-slice multitasking is where the CPU 'switches' its attention between requested tasks from multiple programs. Each task receives the attention of the CPU for a period of time (usually in nanoseconds) before the CPU switches to the next task.

‚è≤Ô∏è The Round-Robin Classroom Analogy

A teacher gives each student exactly 2 minutes to present their work, then moves to the next student regardless of whether they're finished. The teacher uses a timer and strictly enforces the time limit.

Benefit: Everyone gets a fair turn, and no one can monopolize the teacher's attention!

How it works:

  • OS sets a timer interrupt for each process
  • When time quantum expires, OS forcibly switches to next process
  • Processes don't need to cooperate - OS is in control
  • Priority levels can be used to give some processes more CPU time

Advantages: Fair distribution of CPU, responsive system, no single process can monopolize CPU

Disadvantages: More context switching overhead, requires hardware timer support

‚úì Modern Standard: All modern operating systems (Windows 10/11, macOS, Linux) use preemptive multitasking. This provides better stability and responsiveness!

Definition: Multithreading supports the execution of several simultaneous tasks within the same application. A thread is a lightweight process - multiple threads can exist within a single process, sharing the same memory space.

  • The office building = The process (has resources: desks, computers, filing cabinets)
  • The employees = Threads (share the office resources)
  • All employees can access the same filing cabinets (shared memory)
  • Each employee has their own desk and task list (separate execution path)
  • They work on different parts of the same project simultaneously

Processes are executed in the order they arrive. Simple but can lead to the "convoy effect" where short processes wait behind long ones.

Page Fault

A page fault occurs when a process tries to access a page that is not currently in physical memory. The OS must:

  1. Pause the process
  2. Find the page on disk
  3. Load it into a free frame (or replace an existing page)
  4. Update the page table
  5. Resume the process

Thrashing

Thrashing occurs when a system spends more time swapping pages in and out of memory than executing actual processes. This happens when there are too many processes competing for too little physical memory.

üèÉ The Juggling Analogy

Signs of Thrashing:

  • High page fault rate
  • Low CPU utilization despite high demand
  • System becomes extremely slow
  • Disk activity is very high

Solutions:

  • Add more physical RAM
  • Reduce the number of running processes
  • Use better page replacement algorithms
  • Implement working set model

  • Others: 0 (0+0+0) = no access

  • Others: 5 (4+0+1) = read, execute

Part 5: Advanced Concepts

üëë Special Permissions: SUID, SGID, and Sticky Bit ‚ñº

These are advanced permissions that modify how files and directories behave:

SUID (Set User ID) - 4xxx

When set on an executable, the program runs with the owner's permissions, not the user who launched it.

chmod 4755 program
# Now shows: -rwsr-xr-x (note the 's' instead of 'x')

Example: The passwd command needs to modify /etc/shadow (root-only file), so it has SUID set to temporarily run as root.

SGID (Set Group ID) - 2xxx

On executables: runs with the group's permissions. On directories: new files inherit the directory's group.

chmod 2775 shared_folder
# Now shows: drwxrwsr-x (note the 's' in group position)

Use case: Shared project directories where all files should belong to the project group.

Sticky Bit - 1xxx

On directories: only the file owner (or root) can delete files, even if others have write permission to the directory.

chmod 1777 /tmp
# Now shows: drwxrwxrwt (note the 't' at the end)

Example: The /tmp directory - everyone can create files, but you can only delete your own files.

üîê Changing Ownership: chown and chgrp ‚ñº

Beyond permissions, you can change who owns files:

chown - Change Owner

# Change owner only
sudo chown newuser myfile.txt

# Change owner and group
sudo chown newuser:newgroup myfile.txt

# Change recursively (entire directory)
sudo chown -R newuser:newgroup my_folder/

chgrp - Change Group

# Change group only
sudo chgrp newgroup myfile.txt

# Change recursively
sudo chgrp -R developers project_folder/
